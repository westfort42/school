Welcome!

Let's talk about our crazy project:

1. Project Description

Short & sweet summary:

Our project, like everyone else's, is a Residence Management System.

There are currently three types of rooms:

  • Residence halls
  • Apartments
  • Townhouses

There is also a bunch of other stuff to worry about, like meal plans, accomadations, etc. Our task is to design a management system for dealing with these things. Since it was mentioned that our software had to look pretty on mobile and desktop, we decided to go with the ole web browser as our platform (well... and some other stuff — more on that later).

2. Requirement Analysis

1. Quality Requirements

Although it seems straightforward, there are a few things to keep in mind if you've chosen the web as your platform:

  • Security — Communication between the server and the client needs to secure. We'll be using HTTPS (which uses SSL). This is built into a part of our tech stack.
  • Speed — A database query in Java is fast. A database query in JavaScript... not so fast. We need to make sure we keep things feeling fast, whether they actually are or not!
  • Accessibility — Screen sizes that access the web range from 4 inches wide to 80 inches wide, screen resolutions vary from 240p to 4k, and there are also screen readers to consider.

The awesome thing about the web is that there are already easy ways to satisfy these requirements out there, so its just a matter of implementing them in your project.

2. Requirement Analysis

3. Technological Requirements

Tech requirements can get messy quickly when you're targeting the web. However, thanks to the efforts of various open-source communities, it's not so bad anymore. Here's what we think we need so far:

  • Database — We need something with a dead-simple API, that works well with local borwser storage as well as persistent, back-end storage (remember the speed concerns we mentioned?)
  • Server — Our database and server-side logic have to exist somewhere, so we need a basic web server.
  • Client — This is our user interface, which exists in the web browser.
  • Frameworks & dependencies — This is what makes our lives easier and our development time years shorter. We'll be taking advantage of some great open source stuff, and using it everywhere is our tech stack.

We'll be relying heavily on open source stuff, which we'll get to next.

7. Version/Dependency Control

1. Version control — git & SemVer

With any project that uses the incremental model, there are a ton of changes being made to this project by different group members. Typically, this is very hard to keep track of, but thanks to GitHub (and git), we can easily track each members contributions, changes, etc.

GitHub allows you to stage each change to the source code as a 'commit', which is then synced with the master repository. Each commit contains a message explaining the change, shows a diff highlighting what was changed, and allows to reverse the change if necessary. There are also a million other features I won't mention here.

In the spirit of open source software, we use SemVer to dictate the version number. That means that our project won't be at version 1.0.0 when it releases, but something more like 7.2.11 (for example). SemVer versioning is simple; MAJOR.MINOR.PATCH.

7. Version/Dependency Control

2. Dependency control — NPM

NPM stands for 'Node Package Manager'. It is a package repository for JavaScript packages.

In our project, we maintain dependencies (all the open source stuff we used counts as a dependency) through a file called package.json. This file lists the name and version of each package we're using, so everyone running the development branch of the repository can build the app from the same dependencies as everyone else.

To get a development branch up and running, you would run the command $ git clone followed by the name of our master repository. From there, its as simple as running $ npm install to download dependencies. Running the project is done by running $ hoodie start.
